基于统一SQL的大模型友好查询洞察分享

发表于 2026-07-15 更新于 2026-07-23 分类于 学习
本文字数： 2.4k 阅读时长 ≈ 2 分钟

一句话总结

业界跨库 SQL 适配有三条成熟路线（联邦引擎、语义层、引擎统一），均不依赖事后 SQL 方言转换（将统一 SQL 中的函数改写为各库兼容的写法）。当前”统一 SQL + 方言转换”方案存在等价函数依赖的结构性风险，应向语义层直接生成各库 SQL 演进。

背景

五个子域需要跨 StarRocks/Druid/GaussDB 三个异构数据库查询，AI 需生成一种 SQL 让三库都能执行。当前采用”统一 SQL + 函数映射改写”方案，但已发现 Druid 报文场景查询报错的结构性风险。

业界已有多条成熟路线解决跨库 SQL 适配问题，需对比研判正确的演进方向。

当前方案的问题

当前流程：AI 生成统一 SQL（不感知目标库）→ 引擎层函数映射改写 → 分别下推各库执行。

具体表现：

  • StarRocks：原生支持该函数，零改写，执行成功 ✅
  • GaussDB：原生支持该函数，零改写，执行成功 ✅
  • Druid 报文：FROM_UNIXTIME 函数不下推，内存计算报错 ❌
  • Druid SQL：改写为等价函数，执行成功（降级路径）✅

⚠️ 核心问题：生成统一 SQL 后再改写，某库不支持等价函数即失败。方案依赖每个数据库均存在等价函数可映射，新增不支持等价函数的库即失效。

业界三种路线

路线一：联邦引擎——不推 SQL 给源库，引擎层统一执行

Trino 收到查询请求后，不推 SQL 给源库，而是从各库拉取数据到引擎层统一执行 JOIN 和聚合。

✅ 为什么不依赖等价函数：SQL 不在源库执行，源库只需吐数据，不需要支持任何特定函数。

代表实践：Meta 日处理数千万查询、EB 级数据；Uber 15 集群、5000+ 节点、日 50 万查询。

路线二：语义层——定义指标一次，感知目标库自动生成该库 SQL

语义层定义指标（如”CPU 均值 = 按月聚合 CPU 平均值”），感知目标库，自动生成各库专属 SQL。

  • StarRocks → 生成 StarRocks 方言 SQL → 执行 ✅
  • Druid → 生成 Druid 方言 SQL → 执行 ✅
  • GaussDB → 生成 GaussDB 方言 SQL → 执行 ✅

✅ 为什么不依赖等价函数：每个库拿到的是语义层为自己专属生成的 SQL，不存在”映射不了”的问题。

💡 对大模型的友好性：AI 只需声明查什么指标，语义层自动完成方言适配，降低 AI 生成 SQL 的复杂度。

代表实践：dbt MetricFlow、Cube、Netflix DataJunction、Airbnb Minerva。局限：指标建模投入较大（中型企业数百指标 = 数月建模）。

路线三：引擎统一——只用一个引擎，消除方言问题

AI 生成 SQL（只有一种方言，因为只有一个引擎），统一引擎直接执行。

✅ 为什么不依赖等价函数：只有一个引擎、一种方言，不存在跨库映射。

代表实践：Meta 四引擎收敛到 Presto 一个，节省”数十万行代码和数千台机器”；滴滴四引擎收敛到 StarRocks，查询性能 4 倍、总成本降 80%。

⚠️ 关键教训：性能问题可以用工程手段解决，但运维复杂度无法用工程手段消除——这是引擎统一的根本驱动力。局限：迁移成本高，已有异构库存量场景不适用。

选型结论

业界三种路线均不依赖事后 SQL 方言转换，选型如下：

  1. 联邦引擎——不采用。 查询以单数据源为主，不需要跨源 JOIN；联邦引擎拉取数据到引擎层执行，单数据源查询比原生更慢；需额外部署维护 Trino 集群。

  2. 引擎统一——不采用。 三库功能互补（StarRocks 高性能 OLAP、Druid 时序实时、GaussDB 关系型配置），单一引擎无法同时最优；迁移三个生产库业务中断风险大。

  3. 语义层——采用。 三库不动迁移成本最低；感知目标库直接生成该库 SQL，消除等价函数依赖；可与当前方言翻译并行运行，平滑过渡。

方言翻译的结构性风险

当前方案分三步，三个风险均发生在第二步（引擎层函数映射改写）：

  • Step 1：AI 生成统一 SQL（不感知目标库）
  • Step 2：引擎层函数映射改写 ← ⚠️ 三个风险均在此
  • Step 3：改写后的 SQL 下推各库执行

⚠️ 风险① 等价函数依赖：每个库必须存在等价函数可替换，否则改写失败。已发生：Druid 报文无等价函数下推，场景失败。

⚠️ 风险② 语义差异不可处理：仅能做函数名替换，无法处理 NULL 聚合行为、时区默认值、时间精度等语义级差异。

⚠️ 风险③ 维护负担：每个新函数 × 每个新数据库需新增一条映射规则。5 个函数 × 4 个数据源 = 20 条规则，新增 1 个库则变为 25 条。

当前 SQL 方言转换器基于 Apache Calcite 框架实现函数映射。Druid 报文场景失败已验证风险①，风险②③尚未触发但属结构性隐患。

目标架构

核心变化：从”不感知底层 → 统一生成 → 事后改写”到”感知底层 → 直接生成该库 SQL”。

感知机制：语义层维护表-数据库路由映射（metrics 表 → StarRocks、alerts 表 → Druid）。AI 声明指标后，语义层追溯到底层表，查路由映射确定目标库，直接生成该库 SQL。

业界验证：Netflix、Airbnb、Spotify 均已落地。OSI Initiative 40+ 公司参与，语义层正成为产业标准。

总结

  1. 当前”统一 SQL + 方言转换”方案存在结构性风险。 它依赖各库存在等价函数，Druid 报文场景失败已验证此风险。业界三种路线均不依赖事后 SQL 方言转换。

  2. 应向语义层感知底层数据库直接生成各库 SQL 演进。 语义层通过路由映射感知底层，生成时就对了，从根源消除”改不动”的风险。三库不动、迁移成本最低，可与当前方案并行过渡。
